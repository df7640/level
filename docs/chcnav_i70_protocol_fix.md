# CHCNav i70 초기화 프로토콜 수정 내역

## 날짜
2026-03-18

## 배경
CHCNav i70 GNSS 수신기에 BT SPP로 RTCM 보정데이터를 전송하여 RTK Fix를 달성하는 것이 목표.
테라에스(LandStar) 앱은 동일 수신기에서 RTK Fix에 성공하지만, 우리 앱은 RTCM을 보내도 i70이 인식하지 않음 (주황색 보정 LED 무반응, diffAge: null).

## 분석 방법
1. Android 개발자 옵션의 **BT HCI 스눕 로그** 활성화
2. 테라에스로 i70 연결 → RTK Fix 확인 → 버그 리포트 생성
3. btsnoop_hci.log에서 BT SPP RFCOMM 데이터 추출
4. 테라에스가 보내는 초기화 명령과 우리 앱의 초기화 명령을 **바이트 단위 비교**

## CHCNav 바이너리 프로토콜 구조

```
Offset  Size  Field
[0-1]   2B    $$ (0x24 0x24) 매직 바이트
[2]     1B    방향: 0x01=요청(태블릿→i70), 0x00=응답(i70→태블릿)
[3]     1B    시퀀스 번호 (0x00~0xFF, 세션마다 독립 증가)
[4]     1B    페이로드 길이 (전체 메시지 길이 = byte[4] + 7)
[5-6]   2B    프로토콜 ID: 항상 0x04 0x11
[7-9]   3B    패딩 (0x00)
[10-11] 2B    명령 클래스/ID (상위)
[12-14] 3B    패딩 (0x00)
[15]    1B    카운터/플래그
[16-17] 2B    내부 데이터 길이
[18-21] 4B    고정 (0x00 0x01 0x00 0x02)
[22+]   var   서브 명령 구조체 (클래스/ID + 파라미터)
[last-4] 4B   터미네이터: 0x09 0x24 0x0D 0x0A
```

### RTCM 릴레이 래퍼 구조
```
[22-23] 0x00 0x32     서브 구조 타입
[24-25] 0x15 0x04     RTCM 릴레이 서브커맨드
[26-27] block_len     (RTCM 프레임 길이 + 4, Big-Endian)
[28-29] 0x00 0x00     예약
[30-31] frame_len     (RTCM 프레임 길이, Big-Endian)
[32+]   RTCM3 프레임  (D3 헤더 + 데이터 + CRC24Q)
```

## 발견한 차이점 (2건)

### 차이 1: 명령 #16 — `04,5a` 출력 형식 설정

**파일**: `lib/services/chcnav_init_data.dart` 39번 줄
**바이트 위치**: byte[29] (파라미터 값)

| | byte[29] | 의미 |
|---|---|---|
| 테라에스 (정상 동작) | `0x01` | 출력 형식 1 |
| 우리 앱 (수정 전) | `0x06` | 출력 형식 6 |
| 우리 앱 (수정 후) | `0x01` | 테라에스와 동일 |

**명령 전체 (수정 후)**:
```
24 24 01 29 23 04 11 00 00 00 00 00 00 00 00 01
00 10 00 01 00 02 00 0f 04 04 00 02 00 [01] 04 5a
00 00 65 72 52 65 09 24 0d 0a
                                        ^^^^ 수정됨
```

### 차이 2: 명령 #24 — `04,54` 보정 파라미터 설정

**파일**: `lib/services/chcnav_init_data.dart` 55번 줄
**바이트 위치**: byte[29] (파라미터 값)

| | byte[29] | 의미 |
|---|---|---|
| 테라에스 (정상 동작) | `0x23` (35) | 보정 파라미터 35 |
| 우리 앱 (수정 전) | `0x05` (5) | 보정 파라미터 5 (다른 값으로 덮어씀) |
| 우리 앱 (수정 후) | `0x23` (35) | 테라에스와 동일 |

**명령 전체 (수정 후)**:
```
24 24 01 3a 23 04 11 00 00 00 04 54 00 00 00 01
00 10 00 01 00 02 00 0e 04 04 00 02 00 [23] 04 54
00 00 65 72 52 65 09 24 0d 0a
                                        ^^^^ 수정됨
```

**원인**: 명령 #22에서 `0x23`을 설정한 후, 명령 #24에서 같은 레지스터(`04,54`)에 `0x05`를 다시 써서 덮어쓴 것. 테라에스는 `0x23` 한 번만 보내고 끝.

## 추가 확인 사항

### 프로토콜 공통점 (정상)
- RFCOMM DLCI: 2, 표준 SPP UUID 0x1101
- 인증/라이선스 교환 없음
- 초기화 명령 간 타이밍: ~50ms 간격
- trailing 바이트 (터미네이터 앞 2~4바이트): 버퍼 잔여 데이터로 프로토콜의 일부가 아님 (세션마다 달라짐)
