## 1. 전체 흐름 한눈에 보기

```
                       🛰️ GPS/GLONASS/BeiDou/Galileo 위성들
                       (고도 약 20,000km에 떠 있음)
                              │
                              │ ① 위성 신호 (전파)
                              ▼
                     ┌──────────────────┐
                     │  GPS 리시버       │ ← 현장 측량자가 폴(막대)에 꽂아 들고 다님
                     │  (CHCNav i70)    │
                     └────────┬─────────┘
                              │ ② 블루투스 (NMEA 데이터)
                              ▼
                     ┌──────────────────┐       ③ 인터넷(LTE/WiFi)
                     │  스마트폰/태블릿   │◄───────────────────────┐
                     │  (측설 앱)        │                         │
                     └────────┬─────────┘                         │
                              │                          ┌────────┴────────┐
                              │                          │ VRS/NTRIP 서버   │
                              │                          │ (국토정보공사 등) │
                              │                          └────────┬────────┘
                              │                                   │
                              │                          ④ RTCM 보정데이터
                              │                          (오차를 줄여주는 핵심!)
                              ▼
                     ┌──────────────────┐
                     │  정밀 좌표 계산    │
                     │  (RTK Fixed)     │
                     │  오차: 1~3cm      │
                     └────────┬─────────┘
                              │
                              ▼
                     ┌──────────────────┐
                     │  화면에 "앞으로    │
                     │  2.3m, 왼쪽 0.5m" │
                     │  처럼 안내        │
                     └──────────────────┘
```

**한 마디로 요약:**
> 위성에서 대략적인 위치를 잡고 → 인터넷으로 받은 보정 신호로 오차를 cm 단위까지 줄인 뒤 → 설계도면의 좌표를 현장에서 찾아간다.

## 2. 등장하는 장비와 시스템

### 2.1 GPS 리시버 (GNSS 수신기)

| 항목 | 설명 |
|------|------|
| **역할** | 하늘의 위성 신호를 수신하여 현재 위치를 계산하는 장비 |
| **예시** | CHCNav i70, i80, i93 (한국에서 많이 쓰는 중국산 측량용 수신기) |
| **일반 GPS와의 차이** | 스마트폰 GPS는 오차 3~10m, 측량용 수신기는 오차 1~3cm (약 **1000배** 정밀) |
| **모양** | 직경 15cm쯤 되는 둥근 원반 모양. 측량용 폴(약 2m 막대) 꼭대기에 꽂아서 사용 |
| **가격대** | 약 300만~1,500만원 |

**왜 스마트폰 GPS로는 안 되나요?**
- 스마트폰은 신호의 "코드(암호)"만 읽어서 위치를 계산합니다 (코드 측위)
- 측량용 수신기는 신호의 "반송파(전파의 파동 자체)"까지 읽어서 위치를 계산합니다 (반송파 측위)
- 이 반송파의 파장이 약 19cm(GPS L1 기준)이므로, 파동의 위상차를 읽으면 cm 단위 측정이 가능한 것입니다

### 2.2 스마트폰/태블릿 (측설 앱)

| 항목 | 설명 |
|------|------|
| **역할** | GPS 리시버와 블루투스로 연결하여 위치 데이터를 받고, 인터넷으로 보정 신호를 받아 리시버에 전달. 최종적으로 현장 안내를 해주는 "두뇌" 역할 |
| **왜 필요** | GPS 리시버 자체에는 인터넷 기능이 없어서, 스마트폰의 LTE/WiFi를 빌려 보정 신호를 가져와야 함 |
| **앱 예시** | LandStar (상용), longitudinal_viewer_mobile (우리 앱) |

### 2.3 VRS/NTRIP 서버

| 항목 | 설명 |
|------|------|
| **역할** | 전국에 깔려 있는 고정 기준국 네트워크의 데이터를 모아서 "보정 신호"를 만들어 주는 서버 |
| **운영기관** | 국토정보공사(NGII), SKT TMAP VRS, 각 측량회사 등 |
| **비유** | 네비게이션의 "실시간 교통정보"와 비슷. 교통정보가 있으면 더 정확한 도착시간을 알 수 있듯, 보정 신호가 있으면 더 정확한 위치를 알 수 있음 |
| **무료 여부** | 국토정보공사(VRS)는 가입 후 무료 사용 가능 |

### 2.4 GNSS 위성

현재 하늘에는 4개 나라/지역의 위성 시스템이 떠 있습니다:

| 위성 시스템 | 국가 | 위성 수 | 측설 앱에서의 표기 |
|-------------|------|---------|-------------------|
| **GPS** | 미국 | ~31기 | GP |
| **GLONASS** | 러시아 | ~24기 | GL |
| **BeiDou** | 중국 | ~35기 | GB |
| **Galileo** | 유럽 | ~24기 | GA |

측량용 수신기는 이 4개 시스템의 위성을 **동시에** 수신합니다. 보통 한 번에 20~40개 위성을 잡으며, 위성이 많을수록 정밀도가 올라갑니다.

## 3. 1단계: 블루투스로 GPS 리시버 연결하기

### 3.1 무엇이 일어나는가

측설을 시작하려면 먼저 스마트폰 앱과 GPS 리시버를 연결해야 합니다. 이 연결은 **블루투스**로 이루어집니다. 무선 이어폰을 핸드폰에 연결하는 것과 원리가 같습니다.

### 3.2 연결 과정 (상세)

```
[앱에서 "연결" 버튼 터치]
    │
    ▼
① 페어링 (Pairing)
    - 스마트폰이 "CHCNav-i70-XXXX" 같은 이름의 기기를 찾음
    - 최초 1회 페어링 필요 (이후에는 자동)
    - 타임아웃: 약 30초
    │
    ▼
② 소켓 연결 (SPP 프로토콜)
    - 블루투스 SPP(Serial Port Profile) 사용
    - UUID: 00001101-0000-1000-8000-00805f9b34fb (표준 시리얼 포트)
    - 비유: 블루투스로 가상의 "시리얼 케이블"을 만드는 것
    │
    ▼
③ 초기화 명령 교환
    - 앱이 수신기에게 "너 뭐야?" 질문 (디바이스 코드, 펌웨어 버전 조회)
    - 수신기가 "나는 i70이고, 펌웨어 3.2.1이야" 응답
    - 앱이 수신기에게 "NMEA 데이터를 이런 주기로 보내줘" 설정
    │
    ▼
④ 데이터 스트림 시작
    - 수신기가 1초에 1~5회씩 위치 데이터를 앱으로 전송 시작
    - 데이터 형식: NMEA 문장 (텍스트) 또는 바이너리 프레임
```

### 3.3 연결 실패 시 자동 복구

현장에서 블루투스가 끊어지면 작업이 중단되므로, 앱은 **자동 재연결** 기능을 갖추고 있습니다.

| 항목 | 값 |
|------|-----|
| 재연결 시도 간격 | 5초 |
| 최대 재시도 횟수 | 3회 (총 약 15초) |
| 최대 재연결 윈도우 | 300초 (5분) |
| 읽기 버퍼 크기 | 4,096 바이트 |

### 3.4 CHCNav 바이너리 프로토콜 (0xFA 프레임)

CHCNav 수신기는 표준 NMEA 외에 **자체 바이너리 프로토콜**도 사용합니다. NTRIP에서 받은 보정 데이터를 수신기에 전달할 때 이 프로토콜을 씁니다.

```
┌──────┬──────┬──────┬─────┬──────────┬──────┬──────┐
│ 0xFA │ 0xFA │ LEN  │ SEQ │ PAYLOAD  │ CRC  │ 0xDA │
│ 시작 │ 시작 │ 길이 │ 순번│ 실제데이터│ 검증 │ 끝   │
└──────┴──────┴──────┴─────┴──────────┴──────┴──────┘
```

- `0xFA 0xFA`: "지금부터 하나의 메시지가 시작됩니다" 표시
- LEN: 뒤따르는 데이터의 길이
- SEQ: 순서 번호 (0~255 반복). 패킷이 빠지지 않았는지 확인용
- PAYLOAD: 실제 보정 데이터 (RTCM 바이너리)
- CRC: 데이터가 깨지지 않았는지 검증하는 체크섬
- `0xDA`: "메시지 끝" 표시

> **비유**: 택배를 보낼 때 상자(0xFA 시작)에 물건(PAYLOAD)을 넣고, 송장(SEQ, LEN)을 붙이고, 밀봉(CRC)한 뒤, 봉인 스티커(0xDA)를 붙이는 것과 같습니다.

## 4. 2단계: 하늘에서 위성 신호 받기

### 4.1 GPS는 어떻게 위치를 아는가? — "삼변측량"

GPS의 기본 원리는 매우 직관적입니다:

```
       🛰️ A                  🛰️ B                  🛰️ C
       (20,200km 상공)         (20,200km 상공)         (20,200km 상공)
        │                      │                      │
        │ "나와의 거리:          │ "나와의 거리:          │ "나와의 거리:
        │  21,345.678km"        │  22,123.456km"        │  20,987.654km"
        │                      │                      │
        ▼──────────────────────▼──────────────────────▼
                          📍 여기!
                     (세 원의 교차점 = 내 위치)
```

1. 각 위성은 자기 위치와 현재 시각을 담은 신호를 지상으로 쏩니다
2. 수신기는 이 신호가 도착하는 데 걸린 시간을 측정합니다
3. 시간 × 빛의 속도 = 위성까지의 거리
4. **3개 이상의 위성**으로부터의 거리를 알면, 삼변측량으로 위치를 계산할 수 있습니다
5. 실제로는 시계 오차 보정을 위해 **최소 4개 위성**이 필요합니다

### 4.2 수신기가 보내주는 데이터 — NMEA 문장

GPS 리시버는 위성 신호를 처리한 결과를 **NMEA-0183**이라는 표준 텍스트 형식으로 앱에 보냅니다. 사람이 읽을 수 있는 형태입니다:

#### GGA — 핵심 위치 정보 (가장 중요!)

```
$GPGGA,123519.00,3723.46587,N,12656.12345,E,4,12,0.8,45.3,M,19.2,M,1.0,0001*47
  │        │          │    │       │      │ │  │  │   │   │  │   │  │    │    │
  │        │          │    │       │      │ │  │  │   │   │  │   │  │    │    └─ 체크섬
  │        │          │    │       │      │ │  │  │   │   │  │   │  │    └─ 기준국 ID
  │        │          │    │       │      │ │  │  │   │   │  │   │  └─ 보정 나이(초)
  │        │          │    │       │      │ │  │  │   │   │  │   └─ 단위(미터)
  │        │          │    │       │      │ │  │  │   │   │  └─ 지오이드 높이(m)
  │        │          │    │       │      │ │  │  │   │   └─ 단위(미터)
  │        │          │    │       │      │ │  │  │   └─ 해수면 기준 높이(m)
  │        │          │    │       │      │ │  │  └─ 수평 정밀도 (작을수록 좋음)
  │        │          │    │       │      │ │  └─ 사용 위성 수 (12개)
  │        │          │    │       │      │ └─ ★ 품질 지표 (4=RTK Fixed!)
  │        │          │    │       │      └─ 동경(E)
  │        │          │    │       └─ 경도 (126도 56.12345분)
  │        │          │    └─ 북위(N)
  │        │          └─ 위도 (37도 23.46587분)
  │        └─ UTC 시간 (12시 35분 19.00초)
  └─ 문장 종류 (GGA = GPS 고정 데이터)
```

**품질 지표(Quality Indicator)가 가장 중요합니다:**

| 값 | 의미 | 정밀도 | 상태 |
|----|------|--------|------|
| 0 | 무효 | - | ❌ 위성 못 잡음 |
| 1 | GPS (단독측위) | 3~10m | ⚠️ 보정 없이 위성만으로 |
| 2 | DGPS | 0.5~2m | 🔶 차분 보정 적용 |
| 4 | **RTK Fixed** | **1~3cm** | ✅ **측량 가능 (최상)** |
| 5 | RTK Float | 10~50cm | 🔶 수렴 중, 조금 기다려야 함 |

> **현장에서의 의미**: 품질이 "4 (RTK Fixed)"가 되어야 측량을 시작합니다. "5 (Float)"이면 아직 정밀도가 부족하니 조금 기다립니다. "1 (GPS)"이면 보정 신호가 안 오고 있는 것이므로 NTRIP 연결을 확인합니다.

#### GSV — 위성 상세 정보

```
$GPGSV,3,1,12,01,45,123,42,03,67,234,45,07,23,045,38,08,89,012,48*7C
                 │   │   │  │
                 │   │   │  └─ 신호 세기 (42 dB-Hz)
                 │   │   └─ 방위각 (123°)
                 │   └─ 고도각 (45°)
                 └─ 위성 번호 (PRN 01)
```

이 정보를 모으면 "스카이플롯"이라는 하늘 지도를 그릴 수 있습니다. 위성이 하늘에 어디 있는지, 신호가 얼마나 강한지 시각적으로 보여줍니다.

#### GSA — 정밀도 지표 (DOP)

```
$GNGSA,A,3,01,03,07,08,11,14,17,19,22,28,30,,1.2,0.7,1.0*28
                                                │   │   │
                                                │   │   └─ VDOP (수직 정밀도)
                                                │   └─ HDOP (수평 정밀도)
                                                └─ PDOP (위치 정밀도)
```

**DOP (Dilution of Precision, 정밀도 저하 지수)**:
- 위성들이 하늘에 **골고루 퍼져있으면** DOP가 낮아지고 정밀도가 좋음
- 위성들이 한쪽에 **몰려있으면** DOP가 높아지고 정밀도가 나빠짐
- PDOP < 2.0이면 우수, < 4.0이면 양호, > 6.0이면 불량

> **비유**: 삼각측량에서 삼각형이 정삼각형에 가까울수록 정확하고, 찌그러진 삼각형이면 부정확한 것과 같은 원리입니다.

### 4.3 왜 일반 GPS로는 cm 단위가 안 되나?

| 방식 | 원리 | 정밀도 | 사용처 |
|------|------|--------|--------|
| **코드 측위** (스마트폰) | 위성 신호의 "암호 패턴" 도착 시간 측정 | 3~10m | 네비게이션, 지도 앱 |
| **반송파 측위** (측량 수신기) | 전파의 "파동(위상)" 자체를 측정 | 1~3cm | 측량, 건설 |

GPS L1 신호의 반송파 파장은 약 19cm입니다. 반송파 측위는 이 19cm 파장의 위상(파동의 위치)을 정밀하게 측정하여, 파장의 1/100 수준인 약 **2mm** 해상도로 거리를 측정합니다.

다만, 반송파 측위에는 "정수 미지수(integer ambiguity)"라는 문제가 있습니다. 수신기가 처음 시작하면 위성까지의 거리가 "N × 19cm + α"라는 것은 알지만, N이 몇인지 모릅니다. 이 N을 정확히 알아내는 과정이 **RTK의 핵심**이며, 보정 신호가 이를 도와줍니다.

## 5. 3단계: NTRIP — 인터넷으로 보정 신호 받기

### 5.1 왜 보정 신호가 필요한가?

위성 신호만으로는 여러 오차가 존재합니다:

| 오차 원인 | 크기 | 설명 |
|-----------|------|------|
| 전리층 지연 | 2~50m | 전파가 지구 상층 대기의 전리층을 통과하면서 느려짐 |
| 대류권 지연 | 2~10m | 수증기, 기압 등에 의한 전파 지연 |
| 위성 궤도 오차 | 1~5m | 위성의 실제 위치와 공표 위치의 차이 |
| 위성 시계 오차 | 0~3m | 위성 내 원자시계의 미세한 오차 |
| 다중경로 | 0~5m | 건물 반사 등으로 신호가 돌아옴 |

이 오차들을 **"가만히 서 있는 기준국"** 데이터를 이용해 상쇄시키는 것이 RTK 보정의 원리입니다.

### 5.2 기준국과 VRS

#### 실제 기준국 (Physical Base Station)

전국 각지에 정확한 좌표를 알고 있는 GNSS 수신기가 고정 설치되어 있습니다:

```
    🛰️ 위성 신호
         │
         ▼
    ┌──────────────┐
    │ 기준국 수신기  │  ← 정확한 좌표를 이미 알고 있음
    │ (건물 옥상)   │     (예: N 37.12345678°, E 127.12345678°, H 45.678m)
    └──────┬───────┘
           │
           │ "지금 위성으로 계산하면 N 37.12345683°인데,
           │  실제 내 위치는 N 37.12345678°이니까
           │  지금 이 지역 위성 오차는 +0.00000005°야"
           │
           ▼
    이 "오차 정보"가 바로 보정 데이터(RTCM)!
```

#### VRS (Virtual Reference Station, 가상기준점)

기준국이 멀리 있으면 보정 정밀도가 떨어집니다 (기준국에서 10km 이상 떨어지면 급격히 나빠짐). 그래서 등장한 것이 VRS입니다:

```
    기준국 A ──┐
              │
    기준국 B ──┼──→ VRS 서버 ──→ 가상 기준국 (내 위치 바로 옆에 있는 것처럼!)
              │      │
    기준국 C ──┘      │
                     │
                     └─ 주변 여러 기준국의 데이터를 종합하여
                        "마치 내 바로 옆에 기준국이 있는 것 같은"
                        보정 데이터를 만들어줌
```

**VRS의 핵심 아이디어**: "내가 서 있는 위치에 가상의 기준국을 만들자!"
- 사용자가 자신의 대략적 위치(GGA)를 서버에 보내면
- 서버가 주변 기준국 데이터를 보간(내삽)하여
- 그 위치에 맞춤형 보정 데이터를 생성하여 보내줌
- 결과적으로 기준국이 내 바로 옆에 있는 것과 같은 효과!

### 5.3 NTRIP 프로토콜 — 보정 데이터를 받는 방법

**NTRIP (Networked Transport of RTCM via Internet Protocol)**은 인터넷으로 RTCM 보정 데이터를 전송하는 표준 프로토콜입니다. 웹브라우저가 HTTP로 웹페이지를 받듯, 측량 앱은 NTRIP로 보정 데이터를 받습니다.

#### 연결 과정 (상세)

```
[측설 앱]                                           [VRS/NTRIP 서버]
    │                                                      │
    │ ① TCP 소켓 연결                                       │
    │─────────────────────────────────────────────────────►│
    │  (IP: vrs.ngii.go.kr, Port: 2101 등)                 │
    │                                                      │
    │ ② HTTP 요청 (로그인 + 마운트포인트 선택)                │
    │─────────────────────────────────────────────────────►│
    │  GET /VRS-RTCM32 HTTP/1.1                            │
    │  User-Agent: NTRIP Client/1.0                        │
    │  Authorization: Basic (아이디:비밀번호를 Base64 인코딩)  │
    │                                                      │
    │ ③ 서버 응답: "ICY 200 OK" (접속 성공!)                  │
    │◄─────────────────────────────────────────────────────│
    │                                                      │
    │ ④ 내 위치(GGA) 전송 → 서버가 VRS 보정데이터 생성       │
    │─────────────────────────────────────────────────────►│
    │  $GPGGA,123519.00,3723.46587,N,12656.12345,E,...     │
    │                                                      │
    │ ⑤ RTCM 보정 데이터 수신 시작 (바이너리 스트림)          │
    │◄─────────────────────────────────────────────────────│
    │  [0xD3][길이][RTCM 메시지][CRC24Q]...                 │
    │  (1초에 수백~수천 바이트)                               │
    │                                                      │
    │ ⑥ 주기적으로 GGA 재전송 (1~10초 간격)                   │
    │─────────────────────────────────────────────────────►│
    │  (내가 이동하면 VRS도 갱신해야 하므로)                   │
    │                                                      │
```

#### NTRIP 접속에 필요한 정보

| 항목 | 예시 | 설명 |
|------|------|------|
| **서버 주소(IP)** | vrs.ngii.go.kr | VRS 서버의 인터넷 주소 |
| **포트** | 2101 | 서버의 문 번호 |
| **마운트포인트** | VRS-RTCM32 | "어떤 종류의 보정 데이터를 받을 건지" 선택. 라디오 채널과 비슷 |
| **아이디/비밀번호** | - | 서버 가입 시 발급받은 인증 정보 |

**마운트포인트 이름의 의미:**
- `VRS-RTCM32`: VRS 방식, RTCM 3.2 포맷
- `FKP-RTCM32`: FKP(면보정) 방식, RTCM 3.2 포맷
- `RTCM32-GGA`: RTCM 3.2, GGA 기반

#### NTRIP 연결 상태와 에러

| 응답 코드 | 의미 | 현장에서 할 일 |
|-----------|------|---------------|
| ICY 200 OK | 성공! | 정상 — 보정 데이터 수신 시작 |
| 401 | 인증 실패 | 아이디/비밀번호 확인 |
| 409 | 중복 접속 | 다른 기기에서 같은 계정으로 접속 중. 끊고 재접속 |
| SOURCE_NOT_FOUND | 마운트포인트 없음 | 마운트포인트 이름을 확인 |
| GGA_ERROR | GGA 오류 | GPS 위치가 아직 안 잡혀서 GGA를 보내지 못함. 위성 잡힐 때까지 대기 |

#### 자동 재연결

현장에서는 LTE 신호가 불안정할 수 있으므로, 앱은 NTRIP 연결이 끊어지면 자동으로 다시 연결합니다:

| 항목 | 값 |
|------|-----|
| 초기 대기 | 5초 |
| 재연결 간격 | 10초 |
| RTCM 미수신 타임아웃 | 120초 (2분간 보정 데이터가 안 오면 재연결) |
| GGA 전송 간격 | 1~10초 |

### 5.4 RTCM 데이터 — 보정 신호의 실체

NTRIP로 받는 보정 데이터의 형식이 **RTCM (Radio Technical Commission for Maritime Services)** 입니다. 현재 가장 많이 쓰이는 버전은 **RTCM 3.2**입니다.

```
RTCM 3 메시지 구조:
┌──────────┬──────────┬───────────────────────┬──────────┐
│ 0xD3     │ 10비트   │ 메시지 본문            │ CRC24Q   │
│ (시작)   │ (길이)   │ (보정 데이터)          │ (검증)   │
└──────────┴──────────┴───────────────────────┴──────────┘
```

주요 RTCM 메시지 유형:

| 메시지 번호 | 내용 | 설명 |
|-------------|------|------|
| 1004 | GPS 관측 데이터 (확장) | 기준국이 관측한 GPS 위성별 상세 데이터 |
| 1005 | 기준국 좌표 | 기준국(또는 가상기준국)의 정확한 위치 |
| 1006 | 기준국 좌표 + 안테나 높이 | 위와 같으나 안테나 높이 포함 |
| 1012 | GLONASS 관측 데이터 | 기준국이 관측한 GLONASS 위성별 데이터 |
| 1033 | 수신기/안테나 정보 | 기준국 장비 정보 |
| 1074 | GPS MSM4 | GPS 다중신호 관측 데이터 (최신 포맷) |
| 1084 | GLONASS MSM4 | GLONASS 다중신호 (최신) |
| 1094 | Galileo MSM4 | Galileo 다중신호 (최신) |
| 1124 | BeiDou MSM4 | BeiDou 다중신호 (최신) |

### 5.5 보정 데이터의 여정: 서버 → 앱 → 수신기

보정 데이터가 최종적으로 GPS 리시버에 도달하는 전체 경로:

```
[VRS 서버]
    │
    │ ① 인터넷 (TCP) — RTCM 바이너리 데이터
    │
    ▼
[스마트폰 앱]
    │
    │  앱 내부에서:
    │  - RTCM 데이터 수신량 카운팅 (예: "15.3 KB 수신")
    │  - CRC24Q 검증 (데이터 깨짐 확인)
    │  - CHCNav 프로토콜로 래핑 (0xFA 프레임으로 감쌈)
    │
    │ ② 블루투스 — 0xFA 프레임에 담긴 RTCM 데이터
    │
    ▼
[GPS 리시버]
    │
    │  수신기 내부에서:
    │  - 0xFA 프레임 해제
    │  - RTCM 메시지 파싱
    │  - 기준국 관측값과 자신의 관측값 비교
    │  - RTK 엔진 가동 → 정밀 좌표 계산
    │
    ▼
[정밀 좌표 출력] → GGA 품질 지표가 4(Fixed) 또는 5(Float)로 바뀜!
```

## 6. 4단계: RTK 계산 — cm 단위 정밀 좌표 얻기

### 6.1 RTK의 원리 (간략)

RTK(Real-Time Kinematic)는 두 수신기(기준국 + 이동국)가 **같은 시각에 같은 위성을 관측**한 데이터를 비교하여, 공통 오차를 상쇄시키는 기법입니다.

```
같은 위성 📡
    │               │
    │ 전리층 오차     │ 전리층 오차
    │ (거의 같음!)   │ (거의 같음!)
    ▼               ▼
  기준국           이동국
  (좌표 알고 있음)  (좌표 모름)

  기준국 관측값 - 이동국 관측값 = 공통 오차 상쇄 → 정밀한 차이(벡터) 계산
  기준국 좌표 + 차이 벡터 = 이동국의 정밀 좌표!
```

**핵심 통찰**: 두 수신기가 가까이 있으면 (보통 30km 이내), 전리층/대류권 오차가 거의 같으므로 빼기하면 상쇄됩니다!

### 6.2 Fix와 Float의 차이

RTK 계산에서 가장 중요한 것은 **정수 미지수(Integer Ambiguity) 결정**입니다.

| 상태 | 의미 | 정밀도 | 대기 시간 |
|------|------|--------|-----------|
| **Single** (1) | 보정 없음, 위성만으로 계산 | 3~10m | 즉시 |
| **Float** (5) | 보정 적용 중, 정수 미지수 미확정 | 10~50cm | 보통 5~30초 |
| **Fixed** (4) | 보정 완료, 정수 미지수 확정 | **1~3cm** | Float 이후 수초~수분 |

> **비유**: 자(ruler)로 길이를 재는 것에 비유하면
> - Single = 눈대중으로 "약 30cm" (대충)
> - Float = 자를 대되 눈금 사이라서 "29.5~30.5cm 사이" (애매)
> - Fixed = 정확히 눈금에 맞아서 "30.2cm" (확실!)

### 6.3 보정 나이 (Differential Age)

보정 데이터는 **생선과 같아서 신선도가 중요합니다**:

| 보정 나이 | 상태 | 설명 |
|-----------|------|------|
| 0~5초 | ✅ 최적 | 가장 신선한 보정 데이터 |
| 5~15초 | 🔶 양호 | 아직 쓸만함 |
| 15~30초 | ⚠️ 주의 | 정밀도 저하 시작 |
| 30초 이상 | ❌ 불량 | 보정 효과 급감, NTRIP 연결 확인 필요 |

GGA 문장의 `diffAge` 필드에 이 값이 나옵니다. 보정 나이가 커지면 NTRIP 연결이 끊어졌거나 데이터가 지연되고 있다는 신호입니다.

### 6.4 해 상태 (Solve Status) — LandStar의 18단계 분류

LandStar 앱은 위치 품질을 18단계로 세밀하게 분류합니다 (일반적인 GGA의 0~8보다 훨씬 세밀):

| 값 | 상태 | 의미 | 측량 가능? |
|----|------|------|-----------|
| 0 | NONE | 위치 계산 불가 | ❌ |
| 3 | SEARCH_SAT | 위성 탐색 중 | ❌ |
| 4 | SINGLE | 단독 측위 (보정 없음) | ⚠️ 비정밀 |
| 5 | RTD | 실시간 차분 (DGPS) | 🔶 저정밀 |
| 6 | FLOAT | RTK 부동해 | 🔶 중정밀 |
| 7 | WAAS | SBAS 보강 | 🔶 저정밀 |
| 11 | **FIX** | **RTK 고정해** | ✅ **측량 가능** |

## 7. 5단계: 좌표를 현장에서 찾아가기 (측설)

### 7.1 측량 vs 측설

| 구분 | 측량 (Survey) | 측설 (Stakeout) |
|------|---------------|-----------------|
| **방향** | 현장 → 데이터 | 데이터 → 현장 |
| **질문** | "여기 좌표가 뭐지?" | "이 좌표는 현장 어디?" |
| **비유** | 사진 찍기 (현실을 기록) | 네비게이션 (목적지로 안내) |
| **결과물** | 좌표 데이터 파일 | 현장에 박은 말뚝/표시 |

### 7.2 점 측설의 작동 원리

설계도에 "이 지점에 말뚝을 박아라"라는 좌표 (N: 1234567.890, E: 9876543.210)가 있다고 합시다.

```
[목표점]          현재 내 위치
  X ◄───────── 📍
  (설계 좌표)     (GPS로 실시간 측정 중)

  ┌──────────────────────────┐
  │  목표점까지:              │
  │   앞으로:  2.34 m         │
  │   왼쪽:   0.56 m         │
  │   위로:   0.12 m (높이)   │
  │   2D 거리: 2.41 m        │
  │   방위각:  N 45° 23'      │
  └──────────────────────────┘
```

앱이 하는 계산:

```
1. 목표점 좌표: (Nt, Et) = (1234567.890, 9876543.210)
2. 현재 위치:   (Nr, Er) = (1234565.550, 9876542.650)  ← GPS에서 실시간
3. 차이 벡터:   (ΔN, ΔE) = (2.340, 0.560)

4. 2D 거리 = √(ΔN² + ΔE²) = √(5.476 + 0.314) = 2.41m

5. 전후좌우 변환 (내가 바라보는 방향 기준):
   - 진행 방향(forward) = (cos θ, sin θ)    ← 나침반 or GPS 이동벡터
   - 직각 방향(right)   = (cos(θ+90°), sin(θ+90°))
   - 앞/뒤 = ΔN·cos θ + ΔE·sin θ   ← 점곱(dot product)
   - 좌/우 = ΔN·cos(θ+90°) + ΔE·sin(θ+90°)
```

### 7.3 측설 실전 워크플로

```
① 목표점 선택
   - DXF 도면에서 점을 터치하거나
   - 좌표를 직접 입력

② 접근 (거리 > 5m)
   - 화살표 방향과 거리만 표시
   - "북서쪽으로 15.3m 이동하세요"

③ 미세 조정 (거리 < 5m)
   - "앞으로 2.3m, 왼쪽 0.5m"
   - 허용 오차 원(예: 반경 3cm) 표시

④ 허용 오차 이내 도달
   - 🔔 비프음 or 진동으로 알림
   - "앞으로 0.01m, 오른쪽 0.02m — 허용 범위 내!"
   - 말뚝을 박고 "저장" 버튼

⑤ 다음 점으로 이동
   - 자동으로 다음 측설점으로 전환
```

### 7.4 측설 허용 오차

| 용도 | 일반적 허용 오차 | 설명 |
|------|-----------------|------|
| 건축 기초 | 2~3cm | 기둥, 벽체 위치 |
| 도로 중심선 | 3~5cm | 도로 선형 |
| 종단 측량 | 1~3cm (높이) | 도로 높낮이 |
| 지적 측량 | 3~7cm | 토지 경계 |
| 농업/드론 | 10~30cm | 정밀 농업 |

### 7.5 안테나 높이 보정

측량용 폴 꼭대기에 GPS 수신기가 있으므로, **GPS가 측정하는 위치는 수신기(안테나)의 위치**이지 폴의 바닥(땅 위)이 아닙니다.

```
      ┌───┐ ← GPS 안테나 (여기를 GPS가 측정)
      │   │
      │ 폴 │ ← 높이 약 2.0m
      │   │
      └─┬─┘
        │
   ─────┴───── ← 땅 (여기가 알고 싶은 좌표)

   땅의 높이 = GPS 측정 높이 - 안테나 높이
```

**안테나 높이 측정 방법:**
| 방법 | 설명 | 계산 |
|------|------|------|
| 수직 높이 | 폴 길이를 직접 측정 | APC = 입력값 + 안테나 바닥~위상중심 거리 |
| 경사 높이 | 줄자로 안테나 가장자리까지 측정 | 수직 = √(경사² - 반경²), APC = 수직 + 위상중심 |

> APC (Antenna Phase Center) = 안테나의 "전파를 받는 실제 점". 안테나 꼭대기가 아니라 내부에 있으며, 모델별로 다릅니다.

## 8. 데이터 품질 관리

### 8.1 품질 검사 항목

LandStar는 측점을 저장하기 전에 여러 가지 품질 검사를 수행합니다:

| 검사 항목 | 기준 | 설명 |
|-----------|------|------|
| **해 상태** | Fix (4) 이상 | RTK Fixed가 아니면 경고 또는 차단 |
| **PDOP** | < 6.0 (기본) | 위성 배치가 나쁘면 측량 품질 저하 |
| **위성 수** | ≥ 5개 | 최소 위성 수 미달 시 경고 |
| **보정 나이** | < 15초 | 보정 데이터가 오래되면 정밀도 저하 |
| **수평 정밀도 (HRMS)** | < 3cm | 수평 RMS가 임계값 초과 시 경고 |
| **수직 정밀도 (VRMS)** | < 5cm | 수직 RMS가 임계값 초과 시 경고 |
| **이동 감지** | - | 측량 중 수신기가 움직이면 경고 |

### 8.2 다중 에폭 평균 (Multi-epoch Averaging)

정밀한 측량에서는 한 번의 측정이 아니라 **여러 번 측정의 평균**을 사용합니다:

```
에폭 1: N=1234567.891  ─┐
에폭 2: N=1234567.889   │
에폭 3: N=1234567.890   ├─ 평균: N=1234567.890
에폭 4: N=1234567.892   │  RMS: 0.001m (1mm)
에폭 5: N=1234567.888  ─┘
```

에폭(epoch)은 "1회 측정"을 뜻합니다. 보통 1초에 1~5 에폭을 관측합니다.
- 토포 측량: 1~3초 (3~15 에폭)
- 기준점 측량: 10~180초 (50~900 에폭)

## 10. 자주 묻는 질문

### Q1: 앱에서 "Fix"가 안 뜨고 계속 "Float"인데요?

**가능한 원인과 해결:**
1. **보정 데이터 문제**: NTRIP 연결이 끊어졌을 수 있음 → NTRIP 연결 상태 확인
2. **위성 부족**: 건물/나무에 가려 위성이 적음 → 시야 트인 곳으로 이동
3. **다중경로**: 주변 건물에서 반사된 신호가 간섭 → 건물에서 떨어지기
4. **기준국 거리**: VRS 서버 문제로 기준국이 너무 멀 수 있음 → 보정 나이 확인
5. **수렴 대기**: 처음 켰을 때 1~5분 정도 걸릴 수 있음 → 잠시 대기

### Q2: 보정 나이(Diff Age)가 계속 올라가요

NTRIP에서 새 보정 데이터가 안 오고 있다는 뜻입니다:
1. LTE/WiFi 인터넷 연결 확인
2. NTRIP 서버 접속 상태 확인
3. NTRIP 계정이 만료되지 않았는지 확인
4. 앱의 NTRIP 연결을 끊고 다시 접속

### Q3: 위성 수가 충분한데 PDOP가 높아요

위성이 하늘의 한쪽에 몰려 있으면 개수와 무관하게 PDOP가 높습니다. 시간이 지나면 위성 배치가 바뀌므로 10~20분 후 재시도하면 개선될 수 있습니다.

### Q4: 측설 정밀도를 높이려면?

1. **RTK Fixed 상태 확인** (가장 중요)
2. **다중 에폭 관측**: 1회가 아닌 5~10회 평균 사용
3. **안테나 높이 정확히 입력** (특히 높이 측량 시)
4. **PDOP 낮을 때 작업** (< 2.0 권장)
5. **보정 나이 짧을 때 작업** (< 5초 권장)

### Q5: 비 오는 날에도 측량할 수 있나요?

네, 가능합니다. GPS 전파는 구름과 비를 통과합니다. 다만:
- 폭우 시 약간의 신호 감쇄가 있을 수 있음
- 뇌우 시에는 안전상 야외 측량을 피하세요
- 장비 방수 등급 확인 (CHCNav i70은 IP67 방수)
